# HZB 截幀的兩個 Pass 解析 — Build (下採樣 Depth) vs Occlusion Test

> 適用版本：Unreal Engine 5.7.4-release  
> 問題：HZB 截幀看到兩個 pass，一個是 occlusion test、另一個是下採樣 depth，分別是什麼？

---

## 一、結論速覽

你看到的兩個 pass 是 **生產者** 與 **消費者** 的關係：

| 截幀名稱 | 內部 RDG 事件 | 角色 | 本質 |
|---|---|---|---|
| **下採樣 depth** | `BuildHZB` → `ReduceHZB` / `DownsampleHZB` | **生產者** | 把全解析度深度緩衝降採樣成一條 mip 金字塔（HZB） |
| **occlusion test** | `SubmitHZB` → `TestHZB` | **消費者** | 拿上一步的 HZB 對物件包圍盒做遮擋剔除測試 |

執行順序：**先 Build（下採樣）→ 後 Test（遮擋測試）**。兩者都在 `FDeferredShadingSceneRenderer::RenderHzb()` 內依序觸發（`DeferredShadingRenderer.cpp:487-565`）。

---

## 二、Pass 1 — 下採樣 Depth（HZB Build）

### 2.1 它在做什麼

把 SceneDepth（全解析度深度）逐級降採樣成 **Hierarchical Z-Buffer**：一張帶完整 mip chain 的貼圖，mip N+1 的每個 texel = mip N 對應 2×2 區域的「最遠深度」。

- **FurthestHZB**（必建）：保守的「最遠深度」金字塔 → 用於遮擋剔除（被遮擋判斷）
- **ClosestHZB**（選建）：「最近深度」金字塔 → 用於 SSR/SSGI 等 ray marching 加速

> ⚠️ UE 使用 **Reverse-Z**（近 = 1.0，遠 = 0.0）。所以「最遠深度」其實是取 **最小** device Z。原始碼 `HZB.usf:277`：
> ```hlsl
> FurthestHZBOutput_0[OutputPixelPos] = MinDeviceZ;   // 最遠 = 最小 reverse-Z
> ```

### 2.2 兩條實作路徑

入口：`Engine/Source/Runtime/Renderer/Private/SceneTextureReductions.cpp:118` 的 `BuildHZB()`。

#### (a) Compute 路徑 — `ReduceHZB`（主流）

截幀事件名：`ReduceHZB(mips=[0;3] Furthest) 1024x512`（`SceneTextureReductions.cpp:265`）

- Shader：`FHZBBuildCS` → `HZB.usf:HZBBuildCS`
- **一次 dispatch 產生最多 4 個 mip**（`kMaxMipBatchSize = 4`），透過 groupshared LDS 做階層 reduction（`HZB.usf:290-336`）
- 8×8 thread group，每 thread 用 `GatherRed` 抓 2×2 共 4 個深度，取 min/max
- 若同時要 ClosestHZB，會在同一 dispatch 輸出兩條金字塔
- 可選 `AsyncCompute` 與主圖形佇列並行（`SceneTextureReductions.cpp:259-260`）

```hlsl
// HZB.usf:247-257 — 核心降採樣
float4 DeviceZ = Gather4(ParentTextureMip, ParentTextureMipSampler, BufferUV);
float MinDeviceZ = min(min3(DeviceZ.x, DeviceZ.y, DeviceZ.z), DeviceZ.w);  // 最遠
float MaxDeviceZ = max(max3(DeviceZ.x, DeviceZ.y, DeviceZ.z), DeviceZ.w);  // 最近
```

#### (b) Pixel 路徑 — `DownsampleHZB`（fallback）

截幀事件名：`DownsampleHZB(mip=1) 512x256`（`SceneTextureReductions.cpp:302`）

- Shader：`FHZBBuildPS` → `HZB.usf:HZBBuildPS`
- **一個 fullscreen pass 只產生 1 個 mip**，逐 mip 反覆執行
- 只支援 FurthestHZB（`SceneTextureReductions.cpp:284-285` 有 `check(!bOutputClosest)`）
- 用於不支援 compute HZB 或不支援 texture views 的平台（如部分 OpenGL）

```hlsl
// HZB.usf:344-352 — PS 版只取最遠
float4 DeviceZ = Gather4(ParentTextureMip, ParentTextureMipSampler, BufferUV);
float FurthestDeviceZ = min(min(DeviceZ.x, DeviceZ.y), min(DeviceZ.z, DeviceZ.w));
OutColor = FurthestDeviceZ;
```

選擇邏輯（`SceneTextureReductions.cpp:150`）：
```cpp
bool bUseCompute = bReduceClosestDepth || HZBBuildUseCompute(ShaderPlatform);
```
→ 只要需要 ClosestHZB 或平台允許 compute HZB，就走 `ReduceHZB`；否則 `DownsampleHZB`。

### 2.3 輸出尺寸

```cpp
// SceneTextureReductions.cpp:144-147
HZBSize.X = RoundUpToPowerOfTwo(ViewRect.Width())  >> 1;   // 半解析度起步
HZBSize.Y = RoundUpToPowerOfTwo(ViewRect.Height()) >> 1;
NumMips = floor(log2(max(HZBSize.X, HZBSize.Y)));
```
例如 1920×1080 view → HZB mip0 = 1024×512，往下約 10 個 mip 到 1×1。

## 三、Pass 2 — Occlusion Test（HZB 遮擋測試）

### 3.1 它在做什麼

這是 **`FHZBOcclusionTester`**（`SceneOcclusion.cpp:783`），是 GPU 硬體 Occlusion Query 之外的另一套「**基於 HZB 的軟體遮擋剔除**」。它拿 Pass 1 產生的 FurthestHZB，對候選物件的包圍盒逐一測試是否被遮擋。

入口：`FHZBOcclusionTester::Submit()`（`SceneOcclusion.cpp:946`），截幀事件名 `SubmitHZB` → 內含 `TestHZB`。

### 3.2 演算法（每個包圍盒一個像素）

Shader：`FHZBTestPS` → `HZBOcclusion.usf:HZBTestPS`

```hlsl
// HZBOcclusion.usf:13-42
void HZBTestPS(in float4 SvPosition, out float4 OutColor)
{
    // 1) 讀取此 texel 對應物件的包圍盒中心 + 範圍
    float4 BoundsCenter = BoundsCenterTexture.SampleLevel(...);
    float4 BoundsExtent = BoundsExtentTexture.SampleLevel(...);

    // 2) 視錐剔除
    FFrustumCullData Cull = BoxCullFrustum(BoundsCenter.xyz, BoundsExtent.xyz,
                                           View.TranslatedWorldToClip, ...);

    // 3) 視錐內 + 不跨近平面 → 做 HZB 測試
    if (Cull.bIsVisible && !Cull.bCrossesNearPlane)
    {
        FScreenRect Rect = GetScreenRect(int4(0, 0, HZBViewSize), Cull, 4);
        Cull.bIsVisible = IsVisibleHZB(Rect, true);   // ★ 採樣 HZB 比對深度
    }

    OutColor = Cull.bIsVisible ? 1 : 0;   // 1 = 可見，0 = 被遮擋
}
```

`IsVisibleHZB`（在 `Nanite/NaniteHZBCull.ush`）的邏輯：
- 由包圍盒的螢幕矩形大小，挑選**合適的 HZB mip**（讓矩形落在約 2×2 texel 內）
- 採樣該 mip 的最遠深度
- 比較「包圍盒最近點深度」vs「HZB 最遠深度」
- 若包圍盒整個比 HZB 記錄的最遠表面還遠 → **被遮擋**

### 3.3 資料流與延遲

```
CPU: AddBounds() 累積候選包圍盒        (SceneOcclusion.cpp:831)
  ↓ 上傳到 BoundsCenter/ExtentTexture  (RHIUpdateTexture2D, line 1022)
GPU: TestHZB 全螢幕 pass               (每包圍盒 = 1 texel)
  ↓ AddEnqueueCopyPass 回讀            (line 1057)
CPU: MapResults() / IsVisible(Index)  下一幀讀取結果 (line 840, 882)
```

⚠️ **結果有 1+ 幀延遲**（GPU→CPU readback），所以這是「上一幀可見性」的近似，可能造成快速移動時的 pop-in。`SizeX × SizeY` 的結果貼圖每個 texel 對應一個被測物件。

### 3.4 何時啟用

```cpp
// DeferredShadingRenderer.cpp:550
if (FamilyPipelineState->bHZBOcclusion && ViewState
    && ViewState->HZBOcclusionTests.GetNum() != 0)
{
    ViewState->HZBOcclusionTests.Submit(GraphBuilder, View);
}
```
受 `r.HZBOcclusion` 控制（0 = GPU Occlusion Query，1 = HZB 測試）。

## 四、兩者的關係與完整時序

```
RenderHzb()  (DeferredShadingRenderer.cpp:487)
│
├─ [Pass 1] BuildHZB(ViewId=0)                      ← 你看到的「下採樣 depth」
│    └─ ReduceHZB(mips=[0;3] Furthest) 1024x512     (compute, 一次 4 mip)
│    └─ ReduceHZB(mips=[4;7] Furthest) 64x32
│    └─ ReduceHZB(mips=[8;9] Furthest) 4x2
│       (若 PS 路徑則是多個 DownsampleHZB(mip=N))
│
├─ [Pass 2] SubmitHZB                                ← 你看到的「occlusion test」
│    └─ TestHZB  (對所有候選包圍盒測遮擋)
│    └─ EnqueueCopy (GPU→CPU readback)
│
└─ InstanceCullingOcclusionQuery (若 Nanite/GPU 實例剔除啟用)
```

**因果鏈**：Pass 1 產生 HZB → Pass 2（以及下面其他消費者）使用 HZB。沒有 Pass 1 的金字塔，Pass 2 無法做測試。

## 五、HZB 的其他消費者（不只 Occlusion Test）

下採樣出來的 HZB 是**共用資源**，被多個系統取用（解釋為何即使關閉 HZB Occlusion，Build 仍可能存在）：

| 消費者 | 用途 | 需要的 HZB |
|---|---|---|
| **HZB Occlusion Test** | 物件遮擋剔除 | Furthest |
| **Nanite Culling** | Cluster/Instance GPU 剔除 | Furthest (上一幀) |
| **Lumen Screen Probe / Reflections** | Screen trace 加速 | Furthest + Closest |
| **SSR (ScreenSpaceReflections)** | Ray marching 跳步 | Closest (`HZBTracing.ush`) |
| **SSAO / GTAO** | 水平採樣深度 | Furthest |
| **SSGI** | Screen space GI ray march | Closest |
| **Instance Culling Occlusion Query** | 每實例遮擋 | Furthest (上一幀) |

`bClosestHZB` / `bFurthestHZB` 由 `GetViewPipelineState()` 依當前啟用的功能動態決定（`DeferredShadingRenderer.cpp:502`）。

## 六、截幀辨識速查

| 你在截幀看到的字串 | 對應 | 說明 |
|---|---|---|
| `BuildHZB` | RDG scope | 整個 HZB 建構區塊 |
| `ReduceHZB(mips=[0;3] Furthest) WxH` | Compute dispatch | 下採樣 depth，一次 4 mip（**這是你說的「下採樣 depth」**） |
| `ReduceHZB(... Closest Furthest ...)` | Compute dispatch | 同時建近/遠兩條金字塔 |
| `DownsampleHZB(mip=N) WxH` | Pixel pass | 下採樣 depth 的 PS fallback 版本 |
| `SubmitHZB` | RDG scope | 整個遮擋測試區塊 |
| `TestHZB` | Pixel pass | 包圍盒對 HZB 比對（**這是你說的「occlusion test」**） |
| `BeginOcclusionTests` | Raster pass | 傳統 GPU Occlusion Query（與 HZB 測試二選一） |

## 七、調試與 CVar

```
r.HZBOcclusion              0 = GPU Occlusion Query, 1 = HZB 軟體遮擋測試, 2 = 兩者都跑(對比)
r.HZB.BuildUseCompute       1 = 用 ReduceHZB (compute), 0 = 用 DownsampleHZB (pixel)
r.HZB.Visualize             視覺化 HZB 內容
freezerendering             凍結剔除，觀察 HZB 剔除是否過度
```

截幀分析建議：
- 若只看到 `ReduceHZB` 沒有 `TestHZB` → HZB 只被 SSR/Lumen/Nanite 用，未啟用 HZB Occlusion
- 若 `TestHZB` 的 RT 很小（如 256×256）→ 正常，每 texel 是一個被測物件，不是全螢幕
- `ReduceHZB` 出現在 AsyncCompute queue → 與 BasePass 並行，正常優化

## 八、行動端 / Vulkan SM5 注意事項

接續前面文件（[Lumen on Vulkan SM5](./2026_05_20_00.md)）：

- `HZBBuildUseCompute(SP_VULKAN_SM5_ANDROID)` 通常為 true（Adreno 7xx+ 支援 compute HZB），走 `ReduceHZB`
- HZB 是 Lumen Screen Trace 的前置依賴，**Lumen 啟用時 HZB Build 必定存在**
- `TexCreate_UAV` 需求：compute HZB 需要 UAV 支援，`VULKAN_SM5_ANDROID` 的 `bSupportsComputeFramework=true` 滿足（見 `DataDrivenPlatformInfo.ini:141`）
- HZB 貼圖佔用 tile-based GPU 的 VRAM（非 tile memory，因為是 compute 輸出），約 `HZBSize × 1.33（mip chain）× 2 bytes（R16F/PF_R32_FLOAT）`
