# BeginOcclusionTests 在管線中的作用 — 逐物件硬體遮擋剔除

> 適用版本：Unreal Engine 5.7.4-release  
> 問題：截幀看到的 `BeginOcclusionTests` 在管線中作用是什麼、服務於什麼功能？  
> 前置：[2026_05_20_02.md](./2026_05_20_02.md)（確認這是硬體遮擋查詢，非 HZB 軟體遮擋）

---

## 一、一句話結論

`BeginOcclusionTests` 服務的核心功能是 **per-primitive（逐物件）遮擋剔除**：把完全被其他幾何擋住的物件，在進入 BasePass **之前**就剔掉，省下它們的 vertex + pixel 著色與 draw call 開銷。

---

## 二、跨幀回饋系統（核心機制）

這是一個 **跨幀（frame-to-frame）** 的回饋迴路，分兩半執行：

### 本幀 (Frame N) — 發出查詢

`BeginOcclusionTests` 實際動作（`SceneOcclusion.cpp:465-499` `FOcclusionQueryBatcher::Flush`）：
```cpp
RHICmdList.BeginRenderQuery(BatchOcclusionQuery);        // 開始硬體計數
RHICmdList.DrawIndexedPrimitive(
    IndexBufferRHI,
    /*NumVertices=*/  8 * NumPrimitivesThisBatch,         // 每個包圍盒 8 頂點
    /*NumPrimitives=*/ 12 * NumPrimitivesThisBatch);      // 12 三角形 = 1 個 box
RHICmdList.EndRenderQuery(BatchOcclusionQuery);          // 結束計數
```

1. 把候選物件的 **包圍盒當幾何畫出來**
2. 對 **已渲染好的 SceneDepth** 做深度測試（DepthRead，不寫深度）
3. 硬體 `vkCmdBeginQuery`/`vkCmdEndQuery` 統計 **有幾個像素通過深度測試**
4. 結果存入 query pool，**本幀不讀**

### 下一幀 (Frame N+1) — 消費結果

可見性計算階段讀回上一幀的查詢（`SceneVisibility.cpp:2784-2796`）：
```cpp
FRHIRenderQuery* PastQuery = PrimitiveOcclusionHistory->GetQueryForReading(...);
if (RHIGetRenderQueryResult(PastQuery, NumSamples, true))
{
    uint32 NumPixels = (uint32)NumSamples;
    bIsOccluded = (NumPixels == 0);   // ★ 0 個像素通過 = 完全被擋 = 剔除
}
```

`bIsOccluded` 寫入該 primitive 的可見性 bit，決定它 **是否被加入 draw list**。

---

## 三、服務於哪些下游功能

被判定遮擋的物件，**不會進入**以下所有 geometry pass：

| 下游 Pass | 省下的成本 |
|---|---|
| **BasePass (GBuffer)** | draw call + vertex + pixel 著色（**最主要收益**） |
| **Depth Prepass** | 深度繪製 |
| **Shadow Depth Pass** | 陰影投射繪製（每個光源 × 每個 cascade） |
| **Velocity Pass** | 動態向量繪製 |
| **Custom Depth / Distortion 等** | 其他 geometry pass |

→ 本質作用：**用 GPU 上一幀的真實深度測試結果，告訴 CPU 這一幀哪些 mesh component 不用畫**。

---

## 四、在剔除管線中的位置

```
每幀 CPU 可見性計算 (ComputeViewVisibility):
  1. 距離剔除 (Distance Cull)         便宜
  2. 視錐剔除 (Frustum Cull)          便宜，先做
  3. 遮擋剔除 (Occlusion Cull)        ★ 讀「上一幀」的 query 結果
       └─ FetchVisibilityForPrimitives → bIsOccluded
  4. 為通過的 primitive 建 draw list
  5. 渲染:
       Depth Prepass → BasePass → Velocities → ...
  6. BeginOcclusionTests              ★ 為「下一幀」發出新查詢（你截幀看到的）
```

**注意**：截幀裡 `BeginOcclusionTests` 排在渲染後段（RenderVelocities 之後），因為它需要完整的 SceneDepth 才能測試；而它的**結果要等下一幀**才在步驟 3 被消費。

---

## 五、決策邏輯的細節（保守正確性）

`SceneVisibility.cpp:2756-2842` 的完整決策樹：

| 情境 | 判定 |
|---|---|
| Query 結果 NumPixels == 0 | 遮擋 → 剔除 |
| Query 結果 NumPixels > 0 | 可見，記錄可見像素比例 |
| Query 讀取失敗 | **當作可見**（寧可多畫不可漏畫） |
| 無歷史記錄（新物件） | 當作可見，發查詢 |
| 多幀緩衝無當前 query | 沿用上一幀狀態 (`WasOccludedLastFrame`) |
| `View.bIgnoreExistingQueries` | 強制可見（如剛切視角） |

關鍵原則：**保守** — 任何不確定狀況都判為可見，避免該畫的物件被誤剔（會造成物件消失的明顯瑕疵）。

## 六、跨幀延遲與緩解

### 延遲模型

```
Frame N:    畫 box → query → (不讀)
Frame N+1:  讀 N 的結果 → 決定可見性 → 渲染
```

→ **1+ 幀延遲**（多幀緩衝下可能 2-3 幀，取決於 `NumBufferedFrames`）。

### 副作用與緩解

| 問題 | 緩解手段 | CVar |
|---|---|---|
| 快速轉鏡頭 → 邊緣物件 pop-in | 包圍盒世界空間擴張 | `r.ExpandAllTestedBBoxesAmount` |
| 新進視野物件閃爍 | 新物件 box 額外擴張 + 冷卻 | `GExpandNewlyOcclusionTestedBBoxes*` |
| 螢幕空間邊緣誤剔 | 螢幕空間 box 擴張 | `GExpand*BBoxesScreenSpace` |
| 近距離物件不該測 | 近距離跳過遮擋測試 | `NeverOcclusionTestDistance` |

## 七、與其他剔除系統的區別

| 系統 | 粒度 | 驅動 | 對象 |
|---|---|---|---|
| **BeginOcclusionTests (本文)** | 整個 PrimitiveSceneProxy（mesh component） | CPU（讀 GPU query） | 傳統非 Nanite primitive |
| Nanite GPU Culling | Cluster / Instance | GPU（用 HZB） | Nanite mesh |
| HZB 軟體遮擋 (`TestHZB`) | Primitive bbox | GPU（採樣 HZB） | 傳統 primitive（`r.HZBOcclusion=1`） |
| Frustum Cull | Primitive bbox | CPU | 所有 |
| Distance Cull | Primitive | CPU | 所有 |

**關鍵**：
- 它是 **CPU-driven 的 primitive 級剔除** — CPU 決定哪些 proxy 進 draw list
- 跟 Nanite 的 **GPU cluster 剔除** 是兩條獨立路徑
- 跟你截幀的 `BuildHZB` **沒有直接因果**（BuildHZB 是給 Lumen/SSR/Nanite 用的，見 [02 文件](./2026_05_20_02.md)）

## 八、與你的 Lumen / Nanite 場景的關係

接續 [Vulkan SM5 + Lumen 配置](./2026_05_20_00.md)：

- **若場景大量使用 Nanite**：Nanite 走自己的 GPU HZB cluster 剔除，傳統 occlusion query 主要剔的是**非 Nanite 的傳統 primitive**（如 Skeletal Mesh、半透明、特效、未轉 Nanite 的舊資產）
- **行動端 (Adreno) 考量**：硬體 occlusion query 的 `vkCmdBeginQuery` 在 Tile-Based GPU 上可能造成 pipeline stall 與回讀同步成本
- **替代方案**：既然你已為 Lumen 建好 HZB，可考慮 `r.HZBOcclusion=1` 改用 HZB 軟體遮擋（系統 B），複用金字塔、避免硬體 query stall（詳見 [02 文件第七節](./2026_05_20_02.md)）

## 十、總結

1. **作用**：逐物件遮擋剔除，跳過完全被擋住的 mesh component。
2. **服務功能**：減少 BasePass / Shadow / Velocity 等所有 geometry pass 的 draw call 與著色開銷。
3. **機制**：本幀畫包圍盒測 SceneDepth + 硬體 query 計數 → 下一幀讀結果決定可見性。
4. **位置**：渲染後段發查詢（需 SceneDepth 就緒），下一幀可見性計算時消費。
5. **特性**：CPU-driven、primitive 粒度、1+ 幀延遲、保守正確。
6. **與 BuildHZB 無關**：截幀的 BuildHZB 服務 Lumen/SSR/Nanite，是不同剔除路徑。
