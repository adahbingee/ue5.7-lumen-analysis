# 移除 BasePass Lighting (SceneColor) MRT — 3-RT GBuffer-Only BasePass 改造方案

> 適用版本：Unreal Engine 5.7.4-release  
> 目標：在 [4-RT 改造](./2026_05_19_01.md) 基礎上，進一步將 BasePass slot 0 (SceneColor / Lighting) 解耦，得到 **3-RT pure GBuffer-only BasePass** (A/B/C)  
> 前提：場景無材質 Emissive、無 BasePass 間接照明（全靠 Lumen），無需 BasePass Fog、無 SLW、無 Decal 影響 SceneColor

---

## 一、為什麼 BasePass Slot 0 一直存在

在 `Engine/Source/Runtime/Renderer/Private/SceneTextures.cpp:858`：
```cpp
uint32 RenderTargetCount = 0;
// All configurations use scene color in the first slot.
RenderTargets[RenderTargetCount++] = FTextureRenderTargetBinding(Color.Target);  // ★ 無條件綁定
```

這個 slot 在 GBufferInfo 中標記為 `"Lighting"`（`GBufferInfo.cpp:362`），格式 `GBT_Unorm_11_11_10`。它是 SceneColor 的 RTV，BasePass 在這裡寫入：

| BasePass MRT0 寫入內容 | 觸發條件 | 我們是否需要 |
|---|---|---|
| `Color += Emissive` (BasePassPixelShader.usf:1630) | `GetMaterialEmissive()` ≠ 0 | ❌ 用戶確認不需要 |
| `Color += DiffuseIndirectLighting × DiffuseColor` (line 1349) | `ALLOW_STATIC_LIGHTING` 或 SkyLight 非 Lumen | ❌ Lumen 已托底 |
| `Color += Fogging` (line 2272) | `PROJECT_VERTEX_FOGGING_FOR_OPAQUE` | ❌ 用 deferred fog pass |
| `Out.MRT[0] = LightAccumulator_GetResult(...)` (line 2289) | SLW / 透明 / Substrate | ❌ 行動端不用 SLW |
| Decal `ModifyBasePassOutput` | DBuffer decals 改寫 emissive | ❌ 假設無 emissive 修改型 decal |

→ 所有寫入路徑在嚴格 Lumen-only + DefaultLit-only 場景下都退化為 0。**MRT0 一直在浪費 Tile 記憶體與寫入頻寬**。

---

## 二、Lumen Indirect 與 SceneColor 的關係（關鍵驗證）

從 `Engine/Source/Runtime/Renderer/Private/IndirectLightRendering.cpp:999`：
```cpp
FRDGTextureRef SceneColorTexture = SceneTextures.Color.Target;  // ★ 直接從 SceneTextures 拿
```

Lumen Indirect Lighting 不依賴 BasePass「寫入 SceneColor 的初值」——它使用 `SceneTextures.Color.Target` 作為**獨立的 accumulate target**，自己拉一個 RDG pass 進行：
```
SceneColor += LumenDiffuseGI × DiffuseColor
SceneColor += LumenReflections × SpecularColor
```

因此：**只要 SceneColor 在 Lighting Pass 前被 Clear 到 0**，無論 BasePass 是否寫入 MRT0 都沒有差別。

## 三、改造方案總覽

### 方案 A — CW_NONE 遮罩（低風險，部分節省）

利用 UE 既有的 `TStaticBlendStateWriteMask` 模式（`BasePassRendering.cpp:551` 已有 `CW_NONE` 用例）：
```cpp
DrawRenderState.SetBlendState(
    TStaticBlendStateWriteMask<CW_NONE, CW_RGBA, CW_RGBA, CW_RGBA>::GetRHI());
//                            ^^^^^^^ slot 0 不寫
```
+ `PassParameters->RenderTargets[0].SetLoadAction(ENoAction)` 和 `SetStoreAction(ENoAction)`

**效果**：
- ✅ BasePass PSO 仍然有 4 個 RT binding，但 GPU 知道 slot 0 不可寫
- ✅ 大多數 desktop GPU 會 skip color write，節省 ROP 頻寬
- ⚠️ Adreno/Mali tile-based 渲染器**仍會分配 tile memory slot**（attachment count 沒變）
- ✅ 改動小、回退容易

### 方案 B — 完全解綁（高風險，最大節省）★ 用戶實際需求

**將 `Color.Target` 從 BasePass RT list 移除，slot 0 直接成為 GBufferA**。

```
原 6-RT layout:    [Lighting][A][B][C][Velocity][D][E]
舊 4-RT layout:    [Lighting][A][B][C]
新 3-RT layout:    [A][B][C]                     ← 本方案目標
```

**效果**：
- ✅ Tile memory 真正減少 1 個 attachment (11.11.10 → 約 4 bytes/pixel)
- ✅ Vulkan renderpass 的 colorAttachmentCount 從 4 降為 3
- ❌ 不可逆地失去 BasePass emissive / fog / SLW 等
- ⚠️ 需要新增 `ClearSceneColor` pass 在 Lighting 前

下文以 **方案 B** 為主軸。

## 五、新增獨立 SceneColor Clear Pass

**位置**：`DeferredShadingRenderer.cpp` 在 BasePass 後、Lighting 前。

```cpp
// 在 RenderBasePass() 後立即執行
if (Scene->GetShaderPlatform() == SP_VULKAN_SM5_ANDROID
    && CVarGBufferStripSceneColor.GetValueOnRenderThread() != 0)
{
    AddClearRenderTargetPass(
        GraphBuilder,
        SceneTextures.Color.Target,
        FLinearColor::Black,
        FIntRect(0, 0, ViewExtent.X, ViewExtent.Y));
}
```

**Adreno 優化**：因為 SceneColor 此時為「待寫入」狀態，Clear 配合 Vulkan `loadOp=CLEAR` 不會觸發 tile load。在 Lighting pass 後再 store。

## 七、RDG 安全性深度分析

### 7.1 SceneColor 在後續 pass 的訪問

| Pass | 訪問方式 | 改造後影響 |
|---|---|---|
| Lumen ScreenProbeGather | 採樣 SceneColor 做 history reprojection | ✅ Color.Target 仍存在 |
| Lumen Reflections | 寫入 SceneColor (additive) | ✅ Clear pass 已歸零 |
| DiffuseIndirectComposite | 加法寫入 SceneColor | ✅ |
| Direct Lighting (DeferredLightPixelShaders) | 加法寫入 SceneColor | ✅ |
| Translucency Pass | 採樣 + 寫入 SceneColor | ✅ |
| Skybox/Atmosphere | 寫入 SceneColor | ✅ |
| Post Processing | 採樣 SceneColor | ✅ |

**全部安全** — 因為這些 pass 都是直接讀 `SceneTextures.Color.Target` 作為自己的 RTV，與 BasePass 是否寫入無關。

### 7.2 Vulkan RenderPass Subpass Dependency

UE 5.7 在 Android Vulkan SM5 path **不使用 subpass chaining**（那是 Mobile path 的優化）。BasePass 是獨立的 renderpass，所以變更 attachment count 不會破壞 subpass dependency graph。

**但是**：BasePass 結束時若有 `Color.Target` 的 store action 為 STORE，且後續 pass 又有 LOAD action，會觸發不必要的 tile→VRAM→tile 往返。

**最佳實踐 storeOp/loadOp 設定**：
```
BasePass:
  - GBufferA/B/C: storeOp = STORE (lighting pass 需要)
  - SceneColor:   ★ 不在 BasePass framebuffer 中

SceneColorClearPass (新增):
  - SceneColor:   loadOp = DONT_CARE, storeOp = STORE

LumenIndirect/DirectLighting:
  - SceneColor:   loadOp = LOAD (從 Clear pass 載入)
  - GBufferA/B/C: 採樣 (非 RTV)
```

→ Adreno 在 BasePass 期間 tile memory 只承載 3 個 attachment + depth，**真實節省約 4 bytes/pixel 的 tile memory + write bandwidth**。

### 7.3 PSO Cache 失效

切換 `r.GBuffer.StripSceneColor` 會導致：
- 所有 BasePass material PSO 因 RTV count 變化而需要重新編譯
- DDC key 變化使 shader map 需重新生成
- Cooked PSO Cache 需重新採集

**生產建議**：在 `DefaultEngine.ini` 中固定設定，**永不在 runtime 切換**。

## 八、與 Approach A (CW_NONE) 的選擇矩陣

| 維度 | 方案 A (CW_NONE) | 方案 B (Detach) |
|---|---|---|
| 改動行數 | ~10 | ~150-200 |
| 風險 | 低 | 高 |
| Tile memory 節省 | ❌ (Adreno 仍分配) | ✅ (~4 B/pixel) |
| BasePass 寫頻寬節省 | ✅ (CW_NONE) | ✅ |
| 功能損失 | 無 | Emissive/Fog/SLW |
| PSO 改變 | BlendState only | RTV count |
| 回退難度 | 切 CVar | 切 CVar + 重編 |
| 推薦場景 | 通用優化 | 純 Lumen 行動端 |

**用戶場景符合方案 B 的全部前提**，且 tile memory 是 Adreno 性能關鍵 → **強烈推薦方案 B**。

## 九、可選增強：Approach A + B 混合

若你想保留材質寫 Emissive 的能力但又要省 tile memory，可採用：
1. 在 PC/Desktop 平台保留 SceneColor binding（方案 A 或全保留）
2. 僅在 `SP_VULKAN_SM5_ANDROID` shader platform 啟用方案 B
3. 透過 `FDataDrivenShaderPlatformInfo` 平台旗標分流

實作：
```cpp
// FetchGBufferParamsRuntime 中
const bool bIsMobileLumenPath = (Platform == SP_VULKAN_SM5_ANDROID);
Ret.bHasSceneColor = !bIsMobileLumenPath || (CVarGBufferStripSceneColor.GetValueOnAnyThread() == 0);
```

→ Editor preview 與 PC build 保持完整能力；Android shipping APK 拿到 3-RT 優化。

## 十一、Adreno 實機效能預期

對 Adreno 730/740 (Snapdragon 8 Gen 1/2/3) Tile-Based Renderer 的理論收益：

| 指標 | 4-RT BasePass | 3-RT BasePass | 改善 |
|---|---|---|---|
| Tile RT count | 4 | 3 | -25% tile pressure |
| BasePass Tile Memory | ~16 B/pixel | ~12 B/pixel | -25% |
| BasePass GMEM Write BW | ~12 B/pixel | ~8 B/pixel | -33% |
| BasePass Wallclock (1080p) | 4-6 ms | 3-4.5 ms | ~25% |

> 數據基於 4-RT 改造已落地的假設；實際依場景複雜度與 overdraw 變化。

額外好處：Tile pressure 下降後，Adreno 的 GMEM bin size 可變大，**bin count 下降 → 重複 vertex shader 工作量下降**。

---

## 十二、限制總結

開啟方案 B 後**永久喪失**（不可在 runtime 啟用）：
- ❌ 材質 Emissive Color 輸出
- ❌ BasePass 內 Vertex/Pixel Fog
- ❌ SingleLayerWater
- ❌ Substrate optimized unlit path 與 BasePass forward translucent
- ❌ Static Lighting 與 Lightmap Indirect
- ❌ Decals 修改 BasePass emissive 輸出（DBuffer normal/roughness 仍可）

保留：
- ✅ 完整 DefaultLit BxDF 直接照明（Lighting Pass 處理）
- ✅ Lumen GI / Reflections（獨立 pass）
- ✅ Virtual Shadow Maps
- ✅ TSR / Temporal AA
- ✅ Decals 修改 GBuffer (Normal/BaseColor/Roughness)
- ✅ Sky/Atmosphere（獨立 pass）
- ✅ 後處理鏈

## 十三、推薦的最終 Mobile Lumen 管線 (整合 00 + 01 + 02)

```
Frame Start
  ├─ Z Prepass (with Velocity)             ← r.VelocityOutputPass=0 把 Velocity 移到這
  ├─ ★ ClearSceneColor (新增)              ← 取代原 BasePass 對 SceneColor 的 Clear
  ├─ BasePass 3-RT (A/B/C only)            ← 本方案核心
  │    └─ Out.MRT[0]=A, MRT[1]=B, MRT[2]=C
  ├─ Decals (modify GBuffer normal/etc)
  ├─ AO (SSAO 或省略，Lumen 提供 AO)
  ├─ ★ Lumen Scene Update (SW RT)
  ├─ ★ Lumen Diffuse Indirect → SceneColor
  ├─ ★ Lumen Reflections → SceneColor
  ├─ Direct Lighting (各光源 → SceneColor)
  ├─ Sky/Atmosphere → SceneColor
  ├─ Fog (deferred) → SceneColor
  ├─ Translucency → SceneColor
  └─ Post Processing → BackBuffer
```

最終 GBuffer Layout（與 2026_05_19_01.md 整合）：

| Slot | RT | Format | 用途 |
|---|---|---|---|
| 0 | GBufferA | 10.10.10.2 | Normal + PerObjectGBufferData |
| 1 | GBufferB | 8.8.8.8 | M/S/R + ShadingModelID |
| 2 | GBufferC | 8.8.8.8 sRGB | BaseColor + AO |

**3-RT BasePass，每像素 ~12 bytes tile memory，~8 bytes BasePass write 頻寬。**
