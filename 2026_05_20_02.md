# 更正：截幀中「先 Occlusion Test 後 BuildHZB」的真相 — 兩套不同的遮擋系統

> 適用版本：Unreal Engine 5.7.4-release  
> 觸發：截幀 (Frame 97) 顯示 `BeginOcclusionTests` 在 `BuildHZB(ViewId=0)` **之前**執行  
> 修正對象：[2026_05_20_01.md](./2026_05_20_01.md) 把 "occlusion test" 誤認為 HZB 軟體遮擋 (`TestHZB`)

---

## 一、截幀實際順序（用戶觀察正確）

```
RenderVelocities(Opaque)
HZB                              ← GPU Stat scope
  BeginOcclusionTests           ← ★ 先跑「遮擋測試」
HZB                             ← GPU Stat scope
  BuildHZB(ViewId=0)            ← ★ 後跑「下採樣 depth」
vkQueueSubmit
ComputeLightGrid
LightFunctionAtlasGeneration
SkyAtmosphereLUTs
```

**用戶發現確實成立：遮擋測試在下採樣 depth 之前。** 但這揭示了一個關鍵事實 —— 截幀裡的 `BeginOcclusionTests` 並不是我上一份文件描述的 HZB 軟體遮擋（`TestHZB`），而是**另一套完全獨立的硬體遮擋查詢**。

---

## 二、關鍵更正：這是「兩套不同的遮擋系統」

UE 有 **兩套互斥** 的遮擋剔除機制：

| | 系統 A：硬體遮擋查詢 | 系統 B：HZB 軟體遮擋 |
|---|---|---|
| 截幀名稱 | **`BeginOcclusionTests`** | `SubmitHZB` → `TestHZB` |
| 測試對象 | 對 **SceneDepth 深度緩衝** 畫包圍盒 | 對 **HZB 金字塔** 採樣比對 |
| 機制 | `vkCmdBeginQuery` 數通過的像素 | Pixel shader 比對深度，輸出 0/1 |
| 是否需要 HZB | **否** | **是**（依賴 BuildHZB） |
| 控制 CVar | `r.HZBOcclusion=0`（預設） | `r.HZBOcclusion=1` |
| 你的截幀 | ✅ 出現 | ❌ 沒出現 |

**你的截幀裡是系統 A（硬體遮擋查詢），沒有系統 B。** 所以「occlusion test 先跑」是因為它根本不需要 HZB。

> ⚠️ 上一份文件 [2026_05_20_01.md](./2026_05_20_01.md) 把 occlusion test 解釋為 `TestHZB`（系統 B），那是在 `r.HZBOcclusion=1` 時才會出現的 pass。你的專案用的是預設的系統 A，所以看到的是 `BeginOcclusionTests`。

## 三、`BeginOcclusionTests` 到底做什麼（系統 A）

### 3.1 本質：硬體 Occlusion Query

`SceneOcclusion.cpp:465-499` 的 `FOcclusionQueryBatcher::Flush`：
```cpp
RHICmdList.BeginRenderQuery(BatchOcclusionQuery);        // 開始計數
RHICmdList.DrawIndexedPrimitive(
    IndexBufferRHI,
    /*NumVertices=*/  8 * NumPrimitivesThisBatch,         // 每個包圍盒 8 頂點
    /*NumPrimitives=*/ 12 * NumPrimitivesThisBatch);      // 12 三角形 = 一個 box
RHICmdList.EndRenderQuery(BatchOcclusionQuery);          // 結束計數，回讀通過像素數
```

流程：
1. 把候選物件的 **包圍盒當幾何畫出來**
2. 對 **已渲染好的 SceneDepth** 做深度測試（Depth-Read，不寫深度）
3. 用 `vkCmdBeginQuery`/`vkCmdEndQuery` 統計**有幾個像素通過深度測試**
4. 通過像素數 = 0 → 物件完全被遮擋 → 下一幀剔除

RT 綁定（`SceneOcclusion.cpp:1515`）：
```cpp
PassParameters->RenderTargets.DepthStencil = FDepthStencilBinding(
    OcclusionDepthTexture,                          // ← 場景深度（或下採樣的 SmallDepth）
    ERenderTargetLoadAction::ELoad,                 // ← Load 既有深度
    ERenderTargetLoadAction::ELoad,
    FExclusiveDepthStencil::DepthRead_StencilWrite);// ← 只讀深度
PassParameters->RenderTargets.NumOcclusionQueries = NumQueriesForBatch;  // ← 觸發 query pool
```

### 3.2 為何能在 BuildHZB「之前」跑

因為它測試的是 **SceneDepth**，而 SceneDepth 在這個時間點已經完備：
```
Depth Prepass → BasePass → RenderVelocities(Opaque) → [深度已最終] → BeginOcclusionTests
```
硬體遮擋查詢只要深度緩衝就緒就能跑，**完全不依賴 HZB**。所以排在 BuildHZB 前面是正確且合理的。

### 3.3 可選的「下採樣深度」前置（不是 BuildHZB）

注意 `RenderOcclusion` 裡還有一個**容易混淆的下採樣**（`SceneOcclusion.cpp:1465-1479`）：
```cpp
if (UseDownsampledOcclusionQueries())
{
    OcclusionDepthTexture = SceneTextures.SmallDepth;       // 1/4 尺寸深度
    AddDownsampleDepthPass(GraphBuilder, View, SceneDepth, SmallDepth, EDownsampleDepthFilter::Max);
}
```
這是把 SceneDepth 降到 **SmallDepth**（給硬體遮擋查詢用的小深度緩衝），用 `r.DownsampledOcclusionQueries` 控制。**它跟 BuildHZB 是兩回事**：
- `AddDownsampleDepthPass` → 給 HW Occlusion Query 的低解析度深度（單張，非金字塔）
- `BuildHZB` → 給 Lumen/SSR/Nanite 的 mip 金字塔

你的截幀若沒看到這個 downsample，表示 `r.DownsampledOcclusionQueries=0`，遮擋查詢直接用全解析度深度。

## 四、那 `BuildHZB` 是給誰用的？

既然你的遮擋走系統 A（硬體查詢），那截幀裡的 `BuildHZB(ViewId=0)` **不是為了遮擋測試**，而是給其他消費者：

| 消費者 | 為何需要 HZB |
|---|---|
| **Lumen Screen Probe / Reflections** | Screen-space ray marching 跳步加速 |
| **SSR** | 反射 ray march 跳步 |
| **Nanite Culling** | 下一幀 cluster/instance GPU 剔除（用本幀 HZB） |
| **SSAO / GTAO** | 水平深度採樣 |

→ 你跑 Lumen，所以 `BuildHZB` **必然存在**，即使遮擋系統用的是硬體查詢。這就是為什麼截幀同時有「遮擋測試」與「下採樣 depth」兩個看似相關、實則服務不同目的的 pass。

## 五、完整因果鏈（修正版）

```
RenderOcclusion()  (DeferredShadingRenderer.cpp:1445)
│  RDG_GPU_STAT_SCOPE(HZB)   ← 整塊掛在 "HZB" GPU stat 下（命名沿用，易誤導）
│
├─ [可選] AddDownsampleDepthPass → SmallDepth   (給硬體查詢，非金字塔)
│
├─ [系統 A] BeginOcclusionTests                  ← 你看到的「occlusion test」
│    └─ 畫包圍盒 + BeginRenderQuery，對 SceneDepth 測試
│    └─ 不需要 HZB，故排在前面
│
└─ RenderHzb()  (DeferredShadingRenderer.cpp:1539 → 487)
     │  RDG_GPU_STAT_SCOPE(HZB)
     ├─ BuildHZB(ViewId=0)                       ← 你看到的「下採樣 depth」
     │    └─ ReduceHZB / DownsampleHZB 建金字塔
     │    └─ 供 Lumen/SSR/Nanite 使用
     │
     └─ [系統 B，僅 r.HZBOcclusion=1] SubmitHZB → TestHZB
          └─ 你的截幀沒有這個（因為用系統 A）
```

**兩個 `HZB` GPU Stat scope** 的成因：`BeginOcclusionTests` 與 `RenderHzb` 各自開了 `RDG_GPU_STAT_SCOPE(GraphBuilder, HZB)`（分別在 `SceneOcclusion.cpp:1457` 與 `DeferredShadingRenderer.cpp:490`），所以截幀顯示兩個並排的 "HZB" 標籤，第一個裝 OcclusionTests、第二個裝 BuildHZB。

## 六、如何在截幀中區分三者

| 截幀字串 | 系統 | 測試 / 產出對象 | 需要 HZB |
|---|---|---|---|
| `BeginOcclusionTests` | A 硬體查詢 | 對 SceneDepth 畫 box 數像素 | 否 |
| `BuildHZB` / `ReduceHZB` / `DownsampleHZB` | — | 產生 HZB 金字塔 | 否（它**產生** HZB） |
| `SubmitHZB` / `TestHZB` | B 軟體遮擋 | 對 HZB 採樣比對 | 是 |
| `AddDownsampleDepthPass`(EDownsampleDepthFilter::Max) | A 的前置 | 產生 SmallDepth | 否 |

快速判斷你用哪套遮擋：
- 看到 `BeginOcclusionTests` + 無 `TestHZB` → 硬體遮擋查詢（`r.HZBOcclusion=0`）✅ 你的情況
- 看到 `TestHZB` + 無 `BeginOcclusionTests` → HZB 軟體遮擋（`r.HZBOcclusion=1`）

## 七、兩套系統的權衡（給你選型參考）

| 維度 | 系統 A 硬體查詢 (`BeginOcclusionTests`) | 系統 B HZB 軟體 (`TestHZB`) |
|---|---|---|
| 精度 | 高（真實光柵化計數） | 保守（金字塔近似，可能略過剔） |
| GPU 開銷 | 中（畫 box + query） | 低（一個 fullscreen pass） |
| CPU 回讀延遲 | 1-N 幀 | 1+ 幀 |
| 額外依賴 | 深度緩衝 | HZB 金字塔（你已為 Lumen 建好） |
| 行動端 (Adreno) | query 較貴，可能 stall | 較友善，無 query stall |
| 適用 | 桌面 / 主機 | 物件數極多 / 行動端 |

### 行動端建議

你在跑 **Android Vulkan SM5 + Lumen**（見 [2026_05_20_00.md](./2026_05_20_00.md)）：
- HZB 金字塔**已經為 Lumen 建好**（`BuildHZB` 反正會跑）
- 硬體遮擋查詢在 Adreno 上會造成 `vkCmdBeginQuery` 的 pipeline stall 與回讀同步成本
- **可考慮切到系統 B**：`r.HZBOcclusion=1`，複用已建好的 HZB，省下硬體 query 開銷

```ini
[/Script/Engine.RendererSettings]
r.HZBOcclusion=1                  ; 改用 HZB 軟體遮擋，複用 Lumen 已建的金字塔
r.DownsampledOcclusionQueries=0  ; 不再需要 SmallDepth 給硬體查詢
```

切換後截幀順序會變成：`BuildHZB` → `SubmitHZB`/`TestHZB`（遮擋變成在下採樣**之後**，因為它依賴 HZB）。

## 八、結論

1. **你的觀察正確**：截幀中 occlusion test 確實在下採樣 depth 之前。
2. **但要更正定性**：那個 occlusion test 是 **`BeginOcclusionTests`（硬體遮擋查詢）**，不是 [上一份文件](./2026_05_20_01.md) 說的 HZB 軟體遮擋 `TestHZB`。
3. **它能排在前面**，正是因為硬體查詢只依賴 SceneDepth、不依賴 HZB。
4. **截幀的 `BuildHZB` 是為 Lumen/SSR/Nanite 服務**，與這個遮擋測試無直接因果。
5. 若想讓遮擋複用 HZB（行動端可能更省），設 `r.HZBOcclusion=1`，屆時順序會反過來（先 Build 後 Test）。
